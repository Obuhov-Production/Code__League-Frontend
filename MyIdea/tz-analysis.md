## Що змінилось від попереднього аналізу

1. Захист від дублювання реєстрації вже частково є на фронті.
   Після правки `TabTournaments` UI коректніше визначає вже зареєстрований турнір і не повинен слати повторний `POST /api/teams` у нормальному сценарії.

2. Публічний лідерборд реально присутній.
   Є окрема сторінка `/leaderboard` і інтеграція з `GET /api/public/tournaments` та `GET /api/public/leaderboard/:tournamentId`.

3. Пункт про “публічну сторінку турніру” був завищений.
   Публічний leaderboard є, але окремої public tournament page у роутингу не видно.

4. Оголошення не можна ставити як повністю відсутні.
   На фронті є API-хук і rendering announcement-повідомлень у чаті, але немає завершеного admin UI.

## Оновлені пріоритети

### Критично

1. Додати UI для submissions від команди.
2. Додати окремий CRUD раундів / задач для organizer/admin.
3. Додати керування розподілом submissions між jury.

### Важливо

4. Додати деталізацію leaderboard по критеріях.
5. Додати public block зі списком турнірів на landing page або окрему public tournament page.
6. Додати “Ваш сабміт” у `TabOverview`.
7. Додати фронтову перевірку унікальності email у складі команди.

### Бажано

8. Доробити повноцінний UI для announcements.
9. Додати realtime leaderboard.
10. Додати архів турнірів і календар дедлайнів.

## Відомі залежності від бекенду

### 1. Реакції в чаті

Фронт уже підтримує:
- `getChatReactions(room)`
- `socket.emit('react', ...)`
- `reaction:update`

Якщо реакції зникають після reload, проблема майже напевно в тому, що бекенд не зберігає їх у БД.

### 2. Публічний leaderboard

Фронт уже готовий до:
- `GET /api/public/tournaments`
- `GET /api/public/leaderboard/:tournamentId`

Перемикач:

```env
VITE_LEADERBOARD_FROM_BACKEND=false
```

### 3. Остаточний захист від дублювання реєстрації

На фронті UI-захист уже є, але реальний захист має залишатися на бекенді через обмеження/перевірку перед `POST /api/teams`.
